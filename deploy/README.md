# 다중 인스턴스 배포 예시 (Nginx 로드밸런싱)

백엔드를 **2대**로 늘리고 그 앞에 **Nginx**를 두어 부하를 분산하는 예시 구성입니다.

```
        ┌──────────────┐
        │  Nginx (:80) │  ── 라운드로빈
        └──────┬───────┘
         ┌──────┴──────┐
   ┌─────▼────┐   ┌─────▼────┐
   │ backend1 │   │ backend2 │   (Spring Boot, :8080)
   └─────┬────┘   └─────┬────┘
         └──────┬───────┘
          ┌─────▼─────┐
          │  MySQL 8  │
          └───────────┘
```

## 왜 인스턴스를 늘려도 안전한가

동시성 제어를 **DB 비관적 락(`SELECT ... FOR UPDATE`)** 으로 구현했기 때문입니다. 락이
애플리케이션(JVM) 안이 아니라 **DB 행**에 걸리므로, 백엔드가 몇 대든 동일 상품 주문은
같은 행 락 위에서 직렬화됩니다.

- 만약 `synchronized`나 JVM 로컬 락이었다면, 인스턴스가 2대가 되는 순간 각 JVM이 별도 락을
  가져 **중복 주문이 뚫립니다.**
- 이 안전성은 `backend/src/test`의 `OrderConcurrencyTest`(동시 30건 → 1건만 성공)와
  `OrderLockTimeoutTest`(락 점유 시 O012·409)로 검증되어 있습니다. 두 인스턴스가 같은 MySQL을
  바라보는 이 구성에서도 동일하게 성립합니다.

## 실행

```bash
docker compose -f deploy/docker-compose.yml up -d --build
# → http://localhost/api/...  (Nginx가 backend1/backend2로 분산)
```

> ⚠️ 이 구성 파일은 표준 패턴으로 작성한 **예시**이며, 작성 환경에 Docker 데몬이 없어
> 로컬에서 `up` 실행 검증은 하지 않았습니다. 실제 사용 시 환경변수(`DB_PASSWORD` 등)와
> 스키마 초기화 방식(데모는 `ddl-auto=update`, 운영은 마이그레이션 도구 권장)을
> 환경에 맞게 조정하세요.
